# Interview Assistant

> Practice AI system-design interviews with an LLM that scores your
> plan against a structured rubric, then coaches you on what
> to fix — and watches you implement when the question is a
> small buildable agent.

---

## Why it's interesting

The editor and the score are the boring parts. The interesting parts
are the guardrails that keep the LLM honest and the system observable
under cost, retries, and partial failure.

### Capabilities at a glance

- **Pose a real system-design question** (`POST /questions`); the
  app classifies its kind (`traditional_design` /
  `agentic_design` / `agentic_build`) and loads a matching rubric.
- **Write `plan.md`** in a Monaco editor with embedded Mermaid
  diagrams; autosave snapshots every five minutes.
- **Ask for hints** during the session — a Socratic chatbot that
  nudges you toward the next decision without giving the answer.
- **Get a rubric-grounded score** with per-signal verdicts, written
  feedback, and concrete next actions.
- **Read post-eval coaching** generated by two parallel LLM calls:
  a deep-dive senior-engineer reflection, and per-signal inline
  coach blocks beside each gap.
- **For buildable questions**: run `mentor watch` while you
  implement. File saves and Claude Code conversation turns stream
  to the backend; a second eval scores the build phase against a
  different rubric.

### Engineering highlights

- **Forced tool-call output** so the model can't invent signal IDs,
  drop required signals, or emit malformed JSON.
- **Evidence validator** ground-checks every quoted snippet against
  the candidate's own `plan.md` and chat history; ungrounded
  evidence auto-downgrades the verdict.
- **Deterministic score** — verdicts come from the LLM, the
  numeric score from a threshold table. `cannot_evaluate` signals
  drop out of both numerator and denominator.
- **Per-attempt seniority + mode calibration** — same plan judged
  differently for a Junior vs Staff candidate; different rubric
  variant for "build a counter" vs "design Twitter."
- **Atomic conditional updates + idempotency keys** on the CLI
  ingest path so retried event batches dedupe at the DB layer.
- **Retry + per-attempt timeout** on every LLM call (LlmService
  wraps the provider with exp-backoff + jitter).
- **Graceful shutdown.** `BackgroundTaskTracker` waits up to 30s
  for fire-and-forget LLM calls before letting Prisma disconnect.
- **Static API call-flow extractor** that walks every controller
  with `ts-morph`, resolves `this.X.Y(...)` via DI constructor
  types, traces to Prisma + external boundaries.
- **Input guardrails module.** Three preset configs (plan / hint /
  question) bound user input by size, trim, and escape literal
  closing-tag injection (`</plan_md>`) before content reaches an
  LLM prompt. Typed `GuardrailRejectedError` flows through NestJS's
  exception filter to a structured HTTP 400 — the frontend reads a
  stable `code` field, and the rejected content never enters a
  log line by construction.

### Run it locally

```bash
# 1. Postgres
brew install postgresql@16 && brew services start postgresql@16
createdb ai_judge

# 2. Backend
cd backend
npm install
cp .env.example .env             # set ANTHROPIC_API_KEY or use claude CLI
npx prisma migrate deploy
npm run start:dev                # http://localhost:3000/api

# 3. Frontend (in a second terminal)
cd frontend
npm install
npm run dev                      # http://localhost:5173

# 4. CLI (optional — only for agentic_build questions)
cd cli
npm install && npm run build && npm link
mentor --help
```

Without `ANTHROPIC_API_KEY` the backend falls back to the local
`claude` CLI (no key needed) or Ollama. See `backend/.env.example`
for the full list of toggles.

### Verifying the eval rubric (regression suite)

```bash
cd backend
npm run eval:plan                # PlanAgent against every plan fixture
npm run eval:build               # BuildAgent against build fixtures
npm run eval:all                 # both + optional mentor passes
```

Per-fixture diffs (score deltas, signal misclassifications, prompt
+ response audit). See `backend/eval-harness/README.md`.

## Background

Built to scratch my own itch as someone preparing for AI-system-
design interviews — a 60-minute timer and an open editor doesn't
give you anything to react to. The first version was just plan-md +
an LLM score. Every subsequent commit was a question the LLM
couldn't answer well, fixed by either tightening the prompt, adding
a guardrail, or restructuring the rubric. The build-phase capture,
the seniority-aware scoring, the post-eval coaching, and the static
analysis tooling all grew out of the same loop: read one bad
evaluation, find one structural reason it was bad, fix it, re-run
the regression suite.
